## 1. Design principles

- **SQLite semantics:** booleans are `integer({ mode: "boolean" })`; timestamps are
  `integer({ mode: "timestamp" })` defaulting to `sql\`(unixepoch())\``. Use `text` for slugs/markdown.
- **One section + one domain (category) per post**, many tags (join table). Type is a constrained
  enum. This mirrors the four taxonomy axes in `01-product-spec.md` Â§3.
- **Markdown is the stored source** (`body_md`); rendered HTML is *derived* (cached via `use cache`,
  not necessarily persisted â€” see Â§6).
- **Per-request Drizzle client â€” never a module-level singleton** (Workers throws *â€śCannot perform
  I/O on behalf of a different requestâ€ť*). Always `getDb()` inside the handler.
- **Reads run directly in Server Components**; **writes run in Server Actions** that re-verify the
  owner (`04-auth.md`) and revalidate cache tags (`02-architecture.md` Â§5).
- **No auth tables by default** â€” auth uses env-secret credentials + JWT (`04-auth.md`). A `users`
  table is an *optional* alternative documented in Â§7.

## 5. Drizzle schema (`lib/db/schema.ts`) â€” shape

```ts
import { sqliteTable, text, integer, primaryKey, index, uniqueIndex } from "drizzle-orm/sqlite-core";
import { sql, relations } from "drizzle-orm";

export const sections = sqliteTable("sections", {
  slug: text("slug").primaryKey(),
  name: text("name").notNull(),
  examCodes: text("exam_codes"),
  blurb: text("blurb"),
  sort: integer("sort").notNull().default(0),
  active: integer("active", { mode: "boolean" }).notNull().default(true),
});

export const domains = sqliteTable("domains", {
  id: integer("id").primaryKey({ autoIncrement: true }),
  sectionSlug: text("section_slug").notNull().references(() => sections.slug),
  exam: text("exam").notNull(),

  sort: integer("sort").notNull().default(0),

export const POST_TYPES = ["concept","cram","lab","troubleshooting","practice-exam","study-guide","journal","project","resources"] as const;

export const posts = sqliteTable("posts", {
  id: integer("id").primaryKey({ autoIncrement: true }),
  slug: text("slug").notNull().unique(),
  title: text("title").notNull(),
  excerpt: text("excerpt"),
  bodyMd: text("body_md").notNull(),
  type: text("type", { enum: POST_TYPES }).notNull(),
  sectionSlug: text("section_slug").notNull().references(() => sections.slug),
  domainId: integer("domain_id").references(() => domains.id),
  exam: text("exam"),
  status: text("status", { enum: ["draft","published"] }).notNull().default("draft"),
  featured: integer("featured", { mode: "boolean" }).notNull().default(false),
  coverImageKey: text("cover_image_key"),

  updatedAt: integer("updated_at", { mode: "timestamp" }).notNull().default(sql`(unixepoch())`),
}, (t) => [
```

## 6. Rendered HTML: derive, donâ€™t (necessarily) store
Render `body_md` â†’ sanitized HTML in a cached function keyed by post id + `updatedAt`
(`02-architecture.md` Â§4.1), tagged `post-<id>`. This keeps the DB lean and always-consistent with
the Markdown. **Optional optimization:** persist a `body_html` cache column written on save to skip
re-rendering on cold caches â€” only add this if profiling shows the render hits the Worker CPU budget
(`06-deployment.md` Â§4). Start without it.

## 7. Optional `users` table (only if password change from admin UI is wanted)
Default auth stores the owner credential in **Worker secrets** (`04-auth.md`) â€” no DB row needed. If
you want to rotate the password from `/admin/settings`, add:
```
users: id pk, email unique, password_hash text, name text, created_at ts
```
â€¦and have `authorize()` read it via `env.DB`. Trade-off: one more table + a settings flow vs. a
`wrangler secret put`. **Recommendation: skip for v1.**

## 8. Slug strategy
- Generate from title: lowercase, strip non-alphanumerics â†’ hyphens, collapse repeats, trim.
- Enforce **unique**; on collision append `-2`, `-3`, â€¦
- Slug is editable in the editor before first publish; **freeze after publish** (changing a published
  slug breaks links â€” if changed, plan a redirect later). Tag/section/type/domain slugs are stable.

## 9. Migrations workflow
```
# 1. edit lib/db/schema.ts
npx drizzle-kit generate           # emits SQL into ./migrations (out dir = migrations_dir)
npx wrangler d1 migrations apply studyblog-db --local    # dev/miniflare
npx wrangler d1 migrations apply studyblog-db --remote   # production
```
- `drizzle.config.ts` `out` **must equal** `wrangler.jsonc` `d1_databases[].migrations_dir` so
  drizzle-kit writes where wrangler applies. (Details in `06-deployment.md`.)
- **Seed both local and remote** (domains + sample posts) â€” local and remote D1 are independent
  stores. Seeds live in `db/*.sql` (see `05-content-taxonomy.md` for the domain rows) applied via
  `wrangler d1 execute studyblog-db --local|--remote --file=db/seed-domains.sql`.

## 10. Coverage-tracker query (the signature feature)
Per-exam coverage = domains that have â‰Ą1 *published* post, over total domains:
```ts
// pseudo-Drizzle: counts per exam
// totals:  SELECT exam, COUNT(*) FROM domains GROUP BY exam
// covered: SELECT d.exam, COUNT(DISTINCT d.id)
//          FROM domains d JOIN posts p ON p.domain_id = d.id AND p.status='published'
//          GROUP BY d.exam
// percent = covered/total (or weight-weighted: SUM(weight) of covered / 100)
```
Wrap in `use cache` + `cacheTag('posts')` so publishing refreshes it. Render as bars on the home
snapshot and cert hubs; the admin dashboard inverts it to show **empty domains** ("write next").
